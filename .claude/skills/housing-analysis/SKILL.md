# 建案分析（housing-analysis）

使用者通常**只會給一個社區或建案名稱**。你的工作是判斷「**這個建案值不值得買**」：先確認建案與其建商，再上網蒐集評價與新聞，最後產出結構化的優劣分析。**建商分析是其中一個重要環節，但不是全部**——還要分析建案本身。輸出一律用**繁體中文**。

## 何時使用

- 使用者給了建案/社區名稱，想知道「這個建案好不好 / 值不值得買 / 風評如何」。
- 使用者問建案或其建商的口碑、施工品質、公設、價格行情、社區管理，或是否常有漏水、糾紛、交屋延遲、客訴。
- 即使使用者沒明講「建案分析」，只要語境是在評估某個建案，就主動使用本 skill。

## 輸入

- 必要：一個社區/建案名稱（例：「東都綠學」「新東琚」）。
- 選用：縣市/區域（有助於消除同名建案的歧義）。

---

## 流程

### 步驟 1：確認建案與基本資料

1. 用 `WebSearch` 查該建案的基本資料：縣市/區域、**所屬建商（起造人/品牌）**、規模（棟數/戶數）、屋齡或預售/新成屋、型態（大樓/透天）、坐落。
   - 來源：591、樂屋、住展、建設官網、實價登錄平台等。
2. **消除歧義**：同名或多個案 → **列出候選或請使用者補充縣市/區域**；查無 → 誠實說明並請補充。**不要臆測。**
3. 記下基本資料與來源連結，作為「建案總覽」。

### 步驟 2：建商分析（必備子環節）

對該建案的建商做分析（這部分等同原本的建商分析）：

1. 確認建商（代銷 ≠ 建商；要看實際起造的建設公司/集團）。建商無法唯一確定時請使用者澄清，不臆測。
2. 蒐集建商**歷年多個建案**的住戶評價與新聞，刻意涵蓋負面：
   - `「<建商> 評價 / PTT / Dcard / 心得」`、`「<建商> 漏水 / 施工品質 / 交屋 延遲」`、`「<建商> 糾紛 / 客訴」`、`「<建商> 新聞 / 訴訟 / 得獎」`
3. 整理出建商簡介、優點、缺點。

### 步驟 3：建案層級分析

比照建商分析的模式，分析**這個建案本身**（依可得資料）：

- **地段與生活機能**：交通、學區、嫌惡設施、生活採買。
- **格局與公設**：格局方正度、坪效、公設比、車位形式。
- **價格 / 行情**：開價與區域行情比較（如有實價登錄資料）。
- **社區管理**：管委會、管理費、住戶素質討論。
- **已知問題**：該案的漏水、施工品質、糾紛、交屋延遲、廣告不實等。

整理出**建案本身的優點與缺點**（每點附佐證與來源）。

### 步驟 4：輸出（固定模板，見下）

---

## 資料來源：務必嘗試台灣社群

除了新聞與房產平台，蒐集建案與建商評價時 **務必嘗試**參考台灣民眾常用社群（真實住戶心得多在這裡）：

- **PTT**：搜 `<建案/建商> PTT`、`<建案/建商> ptt home-sale / 各地方板` 等；可 `WebFetch` 對應討論串。
- **Dcard**：搜 `<建案/建商> Dcard`（房屋板/居家生活板）。
- **Threads**：搜 `<建案/建商> Threads`。
- **Facebook**：搜在地「<地區> 建案 / 買房」社團、建案/建商粉專與留言。

引用社群內容時：**附來源連結、標註這是網路討論**；單一匿名貼文可信度低，需多來源互相印證才當作較可靠。社群查無討論時，明說「社群討論有限」，不要捏造。

## 誠實性規則（必守）

- **每個重要主張都要附來源連結**；區分「查到的事實」與「我的推論」。
- 資料不足、來源矛盾、或只有單一匿名說法時，**明說不確定**，不要寫得像定論。
- **禁止捏造**任何評價、新聞、建案名稱或事件。寧可說「查無」也不要編。
- 區分**建案個案問題**與**建商整體表現**：個案問題不必然代表整個建商，反之亦然——講清楚。
- 注意時效：舊聞（如多年前糾紛）標日期，並說明是否已改善。

## 界線（非目標）

- 這是**公開資訊彙整**，不是投資建議，也不是法律意見——結尾提醒使用者自行查證、現場勘查、看合約。
- 無法識別建案或建商時，誠實回報並請使用者補充，不要硬湊。
